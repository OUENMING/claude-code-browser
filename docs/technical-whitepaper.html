浏览器自动化 + 微信桥的完整技术栈、与行业主流方案的对齐分析、以及可量化的创新点

§1CDP 直接控制 vs 自动化框架

这是整个系统最底层的技术选型——用什么方式操控浏览器。

我们的选择：直接 CDP

通过 chrome.debugger API 发送原始 CDP 命令（v1.3），不经任何中间层
维度CDP 直接 (我们)Playwright / PuppeteerSelenium WebDriver
反爬检测 无 navigator.webdriver 标记，输入走真实管线 Playwright 默认带 automation 标记；Puppeteer 需额外 patch navigator.webdriver=true，几乎所有反爬系统第一时间封
浏览器进程 用户自己的 Edge/Chrome，完整个人资料、Cookie、登录态 独立 browser context，每次要重新登录、过验证码 独立 driver 进程，同上
输入真实性 Input.dispatchMouseEvent + Input.insertText — 和真人完全一样的协议层事件 底层也是 CDP，效果相同。但 page.click() 不传坐标时有降级 WebDriver 协议在浏览器外模拟，可信度低于 CDP 原生事件
API 易用性 需手写 CDP 命令（{"method":"Input.dispatchMouseEvent","params":{...}}） await page.click('.btn') — 一句搞定 driver.findElement(...).click()

行业对齐 与 QoderWork Browser Connector v1.5.1 和 browser-use (Python) 的技术路线一致——都选择原始 CDP 而非包装框架，核心理由是反爬绕过能力。Chrome DevTools MCP 项目（Google 官方，issue #1794）也确认了"多个 AI 工具通过独立 subprocess 连接同一个 CDP endpoint"的架构。

§2元素定位：Ref + WeakRef vs CSS Selector / XPath

AI 需要精准点到某个按钮。怎么"描述"那个按钮？
方案示例SPA 重渲染后可读性(AI)
Ref ID (我们)[ref_48]WeakRef — 元素被移除后自动变 null，不报错，重新 read_page 即可极好 — AI 只需记住编号
CSS Selector.search-box > input[type="text"]类名/结构变了就失效中等 — 需理解 DOM 层级
XPath//div[@class='card']/span[2]同上,且更脆弱差 — 极冗长
ARIA Role + Namerole="searchbox" name="搜索"语义稳定,但非所有元素有 ARIA好 — AI 能理解语义

创新点 Ref + WeakRef + 惰性坐标解析：每个元素在生成时绑一个 WeakRef，只在 AI 实际要点击时才调用 getBoundingClientRect() 获取坐标。这样 SPA 路由切换后元素移动了也能拿到最新位置；被移除的 WeakRef 自动变 null 不会导致空指针错误。这个组合在现有 browser MCP 实现中未见到。

行业标准 WeakRef 是 ES2021 标准，在内存敏感场景（如元素缓存）正确使用。Chrome DevTools Protocol 自身的 DOM.getBoxModel 也用类似惰性机制。

§3合成输入 vs DOM 事件

填表、点击——区别在于是"浏览器认为你在点"还是"网页认为你在点"。

逐字符 ASCII 输入

// 每个 ASCII 字符 = 3 次 CDP 调用 + 20ms 延迟
Input.dispatchKeyEvent({ type: 'keyDown',  key: 'a' })
Input.dispatchKeyEvent({ type: 'char',    key: 'a' })
Input.dispatchKeyEvent({ type: 'keyUp',    key: 'a' })

// 中文 = Input.insertText + 10-30ms 随机延迟
Input.insertText({ text: '中' })
Input.insertText({ text: '文' })
方法事件层小红书/B站/知乎检测？
element.value = "hello"无事件 — React 不知道值变了不触发检测,但表单不起作用
element.dispatchEvent(new InputEvent(...))DOM 层 — 大部分监听器能捕获容易被 isTrusted=false 识破
Input.dispatchKeyEvent (CDP)浏览器内核层 — JS 看到和真键盘一模一样isTrusted=true,无法区分

行业对齐 点击流程 mouseMoved(50ms)→mousePressed(50ms)→mouseReleased 是 Playwright 和 Puppeteer 底层使用的同一套 CDP 方法。加入 50ms 延迟不是技术需要而是行为模拟（真人不可能 0ms 完成按下到释放）。已验证小红书、知乎、B站均未触发验证码。

§4React/Vue 表单填充：Prototype Setter 劫持

问题

React 和 Vue 使用"受控组件"模式——其框架级的 state 管理通过 Object.defineProperty / Proxy 劫持了 input 的 value setter。直接 input.value = "张三" 不触发 React 的 setState,Vue 的 v-model 也不会同步。框架的"内部状态"和 DOM 的"表面值"不同步。

// 原生 setter — 拿到原始 value 属性的 descriptor
const nativeSetter = Object.getOwnPropertyDescriptor(
  HTMLInputElement.prototype, 'value'
).set;

// 用原生 setter 写入 → React 的劫持者收到"真实用户输入"事件
nativeSetter.call(input, '张三');
input.dispatchEvent(new Event('input', { bubbles: true }));
input.dispatchEvent(new Event('change', { bubbles: true }));

行业标准 与 Cypress 和 Testing Library 使用的技术相同——它们也用原生 setter + 事件冒泡来模拟用户输入。不是新发明,但在 CDP 驱动的浏览器自动化中正确执行需要额外注意:先在 CDP 输入管线中触发（让 React 的合成事件系统收到），再补发原生 DOM 事件（让监听原生事件的框架收到）。两套事件都发，保证框架的受控组件更新。

§5Chrome Extension MV3：保活与通信

双轨保活

MV3 强制使用 Service Worker (SW)，SW 在 30 秒无消息后被浏览器回收。保活是每个 MV3 扩展必须解决的工程问题。

方案机制不足
setInterval 25s 每 25 秒写一次 chrome.storage.local，产生"活动"防止 SW 被回收 macOS 合盖或系统挂起时 setInterval 被暂停，SW 会在恢复后被回收
chrome.alarms 0.5min 系统级闹钟——不受 JS 事件循环暂停影响，到点唤醒 SW 最小间隔 30 秒;唤醒后需重新初始化 WebSocket 连接
双轨 (我们) setInterval 25s 防日常回收 + alarms 0.5min 防睡眠后回收 稍多 CPU，但可忽略

创新点 双轨互补策略在现有 MV3 扩展中不常见——大多数只用一个保活方案。已验证:合盖+插电场景下所有浏览器工具持续可用。

WebSocket 通信 (端口 19222)

为什么是 WebSocket 而非 chrome.runtime.sendMessage？因为 MCP Server 是独立 Node.js 进程,不在 Chrome 进程内，和扩展通信需要一个跨进程通道。WebSocket over 127.0.0.1 是最简单的选择:不暴露网络、零配置、全双工。

行业标准 Chrome DevTools Protocol 自身就用 WebSocket (连到 --remote-debugging-port 时暴露的 WS endpoint)。Playwright 的 browserServer.wsEndpoint() 也是 WebSocket。

§6多 Session 共享: Client-Fallback + id 路由 核心创新

这是整个项目最具原创性的架构决策——标准的 stdio MCP 不支持多个 Claude Code 实例共享同一后端资源。

问题:两个 Claude Code,一个浏览器

终端 session 和微信端子 session 是两个独立的 Claude Code 进程,各自按 MCP stdio 协议 spawn 自己的 Browser MCP 进程。两个进程都想 bind 127.0.0.1:19222 → 第二个 EADDRINUSE 失败。

我们怎么解决的

// 进程 A → bind 19222 成功 → Server 模式
[MCP] Server mode: listening on 19222
[MCP] Extension connected (server mode)

// 进程 B → bind 19222 失败 → Client 模式 → 连到进程 A
[MCP] Port 19222 taken, switching to client mode
[MCP] Client mode: connected to shared server 19222

Server 进程 B 发来的 tool_call — 需要 id 路由,否则多个 client 的回复全混

// Client B → Server: tool_call { id: 3, tool: "tabs_context" }
// Server 重写 id → 发给扩展: tool_call { id: "r42", tool: "tabs_context" }
// 路由表记录: "r42" → { clientWs: B的socket, clientId: 3 }
routeTable.set('r42', { clientWs, clientId: 3 });

// 扩展回包 → Server 查表 → 还原 id → 发给 Client B
if (routeTable.has(msg.id)) {
  const { clientWs, clientId } = routeTable.get(msg.id);
  clientWs.send({ ...msg, id: clientId }); // id 还原为 3
}

创新点：单侧 id 重写 路由表只在 Server 进程维护,Client 进程完全无感——它不知道自己的 id 被改过,也不需要知道。这个模式的简洁性在于:Client 进程的代码不需要任何修改,标准 stdio MCP 协议完全兼容。已有文献和现有多 session 共享方案通常建议用 daemon 模式 (launchd/systemd) 或 Unix socket,但我们用纯 WebSocket + id 路由在 stdio MCP 框架内解决了同一个问题。

参考 Chrome DevTools MCP (issue #1794) 讨论了"多个 AI 浏览器工具使用不同 subprocess 并行连接"的架构——他们依赖 Chrome 自身的多客户端支持。我们的情况比他们更复杂:瓶颈不在 Chrome 的 CDP 端口,而在 MCP 进程到 Extension 的 WebSocket 跳。所以我们的路由方案解决的是更下层的问题。

§7Visual Indicator：Shadow DOM 隔离 标准实践

在 AI 操作浏览器时,用户需要看到"哪个元素被点了"、"正在干什么"。直接在页面 DOM 里插 div 会跟网页 CSS 打架（比如页面设了 * { margin:0 }）。

解决：用 Shadow DOM（Web Components 标准,浏览器原生支持）创建一个独立的渲染空间。Shadow root 里的样式完全不受外部 CSS 影响,反之亦然——页面永远看不到高亮框的样式,高亮框也永远不会被页面的 display:none !important 隐藏。

const container = document.createElement('div');
container.id = 'cc-shadow-container';
const shadow = container.attachShadow({ mode: 'open' });
// CSS isolation: zero interference with page styles

行业标准 Shadow DOM 隔离是浏览器扩展 overlay 的标准做法（如 Grammarly、LastPass）。不是新发明但正确应用。

§8MCP 协议: stdio Transport + 工具注册

遵循 Anthropic 官方的 @modelcontextprotocol/sdk (v1.0+),使用 StdioServerTransport 作为传输层。每个 MCP 工具注册包含完整的 JSON Schema 描述:

{
  name: 'computer',
  description: '鼠标/键盘/截图交互。13 种 action...',
  inputSchema: {
    type: 'object',
    properties: {
      action: { type: 'string', enum: [
        'left_click','right_click','double_click',
        'type','screenshot','scroll', ...
      ]},
      ref:     { type: 'string' },
      text:    { type: 'string' },
    },
    required: ['action']
  }
}

行业标准 MCP SDK 的 stdio transport + JSON Schema 工具注册是完全标准的做法，与 Anthropic 官方示例和其他 MCP 服务器（如 Chrome DevTools MCP、Figma MCP）一致。

§12现实与行业实践

合成输入避免检测 —— 使用 CDP 的 Input.dispatchMouseEvent 和 Input.insertText 产生 isTrusted=true 的事件，与 Playwright、Puppeteer、QoderWork 的底层实现一致。小红书、知乎、B站均未触发验证码。
MV3 扩展 + Service Worker —— 遵循 Chrome 最新的扩展规范，chrome.debugger API 是拿到 CDP 控制权的标准方式。双轨保活解决了 SW 30s 回收的工程挑战。
MCP stdio transport —— 遵循 Anthropic 的参考架构。工具注册用的 JSON Schema 格式与 Chrome DevTools MCP 等官方项目一致。
未来演进方向 —— 目前通过 WebSocket 内部桥接，而非纯 MCP stdio。如果要开源或标准化，可以考虑去掉 WebSocket 中间层，让 Extension 直接暴露 MCP 协议的 HTTP endpoint (MCP Streamable HTTP transport)，简化部署。
